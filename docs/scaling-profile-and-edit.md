# 大規模遊戲與收藏的擴充策略（約 8000 款 + 我的收藏）

當**遊戲總數約 8000 款**、**玩家檔案**新增**「我的收藏」**區塊，且單一使用者可能擁有大量喜好/收藏時，下列頁面與模組需要預先考慮載入與渲染策略，避免卡頓、記憶體與 API 負載過大。

---

## 一、現狀與問題整理

| 項目 | 現狀 | 問題（量放大後） |
|------|------|------------------|
| **GameNames** | 一次載入全庫（分批並行 100/頁），localStorage 快取 | 8000 款 ≈ 80 次並行請求、快取體積大、首載慢、行動裝置壓力大 |
| **edit-games-drag 新增遊戲** | `_acGames` 從全庫建清單，輸入時前端 filter 8000 筆取前 12 | 每次輸入都在 8000 筆上做 matchScore，且需先等全庫載完 |
| **edit-games-drag 各區列表** | 每區 `innerHTML = games.map(...)` 一次渲染全部 | 單區數百款 → 數百 DOM + 拖放事件，手機卡頓 |
| **edit-games（手機編輯）** | 每區 `displayXxxGames()` 一次渲染全部標籤 | 同上，列表過長時 DOM 過多 |
| **玩家檔案（喜好）** | 各分頁整批渲染該分頁所有遊戲 | 喜好 + 未來「我的收藏」可能破百、破千，一頁過長、難找 |
| **我的收藏（新區塊）** | 尚未實作 | 設計上即應視為「可能超級多」，需與喜好同一套策略 |

---

## 二、整體策略原則

1. **不要在前端一次持有/渲染 8000 款**  
   搜尋/新增遊戲改為「**輸入關鍵字 → 呼叫 API 搜尋**」，只取前 20～30 筆顯示。
2. **列表區（喜好、收藏、編輯頁各區）**  
   改為「**只渲染前 N 筆（如 24～50）+ 顯示更多 / 分頁**」或「**虛擬捲動**」，避免單一區塊一次產生數百個節點。
3. **GameNames**  
   保留「**已出現在使用者資料裡的遊戲名稱**」的查詢即可（可從 API 只取 user 相關，或延遲載入全庫）；autocomplete 改為依搜尋 API 結果動態顯示，不再依賴全庫 _acGames。
4. **API**  
   現有 `tables/game_database?search=關鍵字&limit=20` 已支援，可直接用於「搜尋遊戲」；若需依「名稱開頭」排序可再補 sort 或專用查詢。

---

## 三、各模組建議做法

### 3.1 遊戲總庫約 8000 款：載入與搜尋

- **GameNames.load()**  
  - **選項 A（建議）**：不再為「新增遊戲 autocomplete」預載全庫。改為：  
    - 僅在「需要顯示名稱/頭像」時，對「當前頁或當前使用者已有的遊戲 ID/名稱」做小範圍載入或按需查詢。  
    - 若現有流程仍依賴全庫 nameMap（例如 formatTag、lookup），可保留「背景分批載入 + 快取」，但改為**非阻塞**、或僅在 recommend/explore 等頁需要時才觸發；edit 頁不強制 await 全庫。
  - **選項 B**：維持全庫快取，但 edit 頁的「新增遊戲」**不**用 _acGames，改為呼叫搜尋 API（見下）。
- **搜尋 API**  
  - 使用既有 `GET tables/game_database?search={關鍵字}&limit=20&page=1`。  
  - 前端：輸入 2～3 字元後 debounce 呼叫，將結果當成 autocomplete 下拉選單來源；選中後再呼叫 GameNames.lookup 或單筆 API 取得顯示用欄位（若尚未在 nameMap）。

### 3.2 edit-games-drag.html（拖放編輯頁）

- **新增遊戲（autocomplete）**  
  - 改為「**搜尋 API 驅動**」：  
    - 移除或不再依賴 `loadAcGames()` 的 8000 筆 _acGames。  
    - 輸入時用 `tables/game_database?search=...&limit=20` 取得候選，只渲染這 20 筆為下拉選單。  
  - 好處：首載不再等全庫、輸入時不掃 8000 筆、手機也順。
- **各區遊戲列表（超喜歡/喜歡/…/沒興趣）**  
  - **方案 A**：每區「**只渲染前 N 筆（例如 50）+ [顯示更多] 按鈕**」。點「顯示更多」可再 append 下一批或展開全部；若單區破百，可考慮分頁（每頁 24）。  
  - **方案 B**：若希望單區可一次看到很多且可拖放，可引入「**虛擬列表**」（只渲染可見區 DOM），拖放時更新資料與順序，再同步到後端。  
  - 建議先做方案 A，實作簡單；若實測仍卡再考慮虛擬列表。
- **拖放**  
  - 若採「前 N + 顯示更多」：拖放僅在「目前已渲染的項目」上操作；未展開的項目可用「移動到…」按鈕或展開後再拖。  
  - 若採虛擬列表：需保留「資料層的順序」與「可見 DOM 的對應」，拖放時更新資料陣列並重算虛擬區間。

### 3.3 edit-games.html（手機編輯頁）

- **列表顯示**  
  - 與 edit-games-drag 一致：每區（喜歡/普通/不喜歡）「**前 N 筆 + 顯示更多**」，避免一次渲染數百個標籤。
- **新增遊戲**  
  - 目前為手動輸入文字，沒有 8000 款下拉問題。  
  - 若未來要改為「選遊戲」：同樣用搜尋 API 取得候選，不要載入全庫。

### 3.4 玩家檔案（profile）：喜好 + 我的收藏

- **喜好（各分頁）**  
  - 與 [profile-future-display.md](./profile-future-display.md) 一致：  
    - 每分頁內「**只顯示前 12～24 張 + 顯示更多／展開**」或分頁（每頁 N 張）。  
  - 資料若單次 API 就回傳該類全部，前端只渲染前 N；「顯示更多」可再取下一批或改為「展開已載入的剩餘」。
- **我的收藏（新區塊）**  
  - 設計上即當作「**可能超級多**」：  
    - 與喜好同一套：**前 N 張 + 顯示更多** 或 **分頁**；若單區破百，可考慮虛擬捲動或分頁。  
  - 後端若「我的收藏」為獨立表或陣列，API 可支援「只回傳前 M 筆 + total」，前端分頁或「載入更多」時再要下一批。
- **錨點 / 區塊導覽**  
  - 區塊變多（喜好、收藏、徽章…）時，可用錨點或手風琴，見 profile-future-display.md。

## 四、實作優先順序建議

| 優先 | 項目 | 說明 |
|------|------|------|
| P0 | edit-games-drag 新增遊戲改搜尋 API | 不再用 _acGames 8000 筆，改 `?search=...&limit=20`，可立即減輕首載與輸入卡頓 |
| P0 | 玩家檔案「喜好」每分頁前 N + 顯示更多 | 避免單分頁上百張一次渲染 |
| P1 | edit-games-drag 各區列表前 N + 顯示更多 | 單區破 50 時即可有感改善 |
| P1 | 「我的收藏」上線時直接做前 N + 顯示更多 | 與喜好同一套，避免一上線就爆量 |
| P1 | edit-games 列表前 N + 顯示更多 | 與 drag 頁一致，手機體驗一致 |
| P2 | GameNames 改為按需/小範圍載入 | 可選：僅在需要 formatTag 的遊戲子集載入，或延遲/背景全庫 |
| P2 | 虛擬列表（編輯頁或 profile） | 僅在「前 N + 顯示更多」仍不足時再考慮 |

---

## 五、API 使用方式摘要

- **搜尋遊戲（autocomplete / 選遊戲）**  
  `GET tables/game_database?search={關鍵字}&limit=20&page=1`  
  回傳 `{ data: [...], total }`，前端只顯示 `data` 作為候選。
- **使用者遊戲資料**  
  維持現有 `getUserById` 等；若未來單一使用者遊戲量極大，可再拆成「各類只回前 M 筆 + total」，前端分頁或「載入更多」再要下一批。

---

## 六、小結

- **8000 款**：不在前端一次載入全庫做 autocomplete；改為**搜尋 API + 前 20 筆**。  
- **編輯頁（drag / 手機）**：各區列表**前 N + 顯示更多**（必要時再虛擬列表）。  
- **玩家檔案**：喜好與**我的收藏**都採用**前 N + 顯示更多**或分頁，並在區塊變多時搭配錨點或手風琴。  
- 依上述優先順序實作，可在大數量下維持可用的載入速度與操作流暢度。
