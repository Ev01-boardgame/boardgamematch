# 登入政策討論：哪些強制登入、哪些不用

## 一、目前狀態（簡要）

| 頁面／功能 | 是否強制登入 | 未登入時行為 |
|------------|--------------|--------------|
| **桌遊探索** (recommend) | 是 | 整頁隱藏，只顯示「請先登入」+ 引導分享喜好 |
| **我的檔案** (profile) | 是 | 顯示「請先登入」 |
| **探索清單** (explore-list) | 是 | 顯示請先登入，才能管理清單 |
| **我的清單** (my-collections) | 是 | 顯示請先登入才能建立主題清單 |
| **遊戲列表** (games-list) | 是 | 無使用者則 showNotLoggedIn |
| **桌遊問答** (quiz) | 否（可瀏覽） | 開始挑戰時檢查，未登入會 alert「請先登入！」 |
| **玩家頁** (player) | 否 | 評價時檢查，未登入會 alert「請先登入才能評價」 |
| **隨機推薦** | 否 | 可選「純隨機」；MBTI／桌遊偏好需登入才有個人化結果 |
| **桌友適性** | 否 | 可做題；「儲存到我的檔案」需登入才顯示／寫入 |
| **首頁、玩家數據、揪桌趣處、貢獻排行、今晚玩什麼、分享喜好** | 否 | 可瀏覽；部分功能需登入（如寫入、個人化） |

---

## 二、方案選項（供討論）

### 方案 A：維持現狀
- **強制登入**：桌遊探索、我的檔案、探索清單、我的清單、遊戲列表。
- **其餘**：可進入頁面，但「寫入／個人化」時再要求登入或提示。
- **優點**：邏輯清楚，需個人資料的頁面才擋。  
- **缺點**：未登入者一進桌遊探索就只看到請先登入，體驗較硬。

### 方案 B：全部不強制登入，但功能分級
- **桌遊探索**：未登入也可進入，可**瀏覽**清單與遊戲卡，但**不能評分**（按評分時提示登入）。
- **我的檔案**：未登入顯示「登入後可看個人檔案與偏好」或簡易說明，不擋進入。
- **探索清單／我的清單／遊戲列表**：未登入可瀏覽；建立／編輯／刪除時再要求登入。
- **桌友適性、隨機推薦、問答、玩家頁**：同現狀，可進入，寫入或個人化時再檢查登入。
- **優點**：新訪客能先「逛」，再決定是否登入；轉換較柔和。  
- **缺點**：需在每個「寫入／個人化」入口加登入檢查與提示，實作稍多。

### 方案 C：全部強制登入
- 進站後若未登入，一律導向登入或首頁登入區，再開放其他頁。
- **優點**：資料完整、邏輯最單純。  
- **缺點**：門檻高，不利新用戶先體驗。

---

## 三、建議（可依產品目標調整）

- 若**希望更多人先體驗再登入**：傾向 **方案 B**（都不強制擋整頁，但寫入／個人化時再要求登入）。
- 若**希望確保資料品質、減少匿名操作**：傾向 **方案 A** 或 **方案 C**。
- **折衷**：桌遊探索改為方案 B（未登入可瀏覽、不可評分）；我的檔案、我的清單、探索清單維持需登入才能「編輯／建立」（可選：未登入仍可進入頁面看說明）。

## 四、方案 B 逐項影響討論（採 B 時可一個一個決定）

以下依「方案 B：全部不強制登入、功能分級」逐頁／逐功能列出：**現狀**、**B 的目標行為**、**影響**（UX／資料／實作／注意）、**待決**。實作前可逐項討論是否照做或微調。

---

### 1. 桌遊探索 (recommend.html)

| 項目 | 說明 |
|------|------|
| **現狀** | 一進入就檢查 `currentUser` 或 `googleUser`，沒有則整頁隱藏 `#game-collections-section`，只顯示「🔒 請先登入」+ 前往分享喜好。 |
| **B 目標** | 未登入也可進入，可**瀏覽**所有清單與遊戲卡；**評分**（喜歡／不喜歡／想玩等）時再檢查登入，未登入則提示「請先登入才能評分」並中斷。 |
| **寫入點** | `rateGame(collectionId, gameName, rating)`（約 1453 行）、隨機推薦的「個人化」模式（需 currentUser）。 |
| **影響** | **UX**：新訪客能先看清單與遊戲、再決定是否登入評分。**資料**：無變動，評分仍僅登入者。**實作**：需改 init 流程——未登入時仍載入 collections + 遊戲並顯示，僅不載入/不依賴 currentUser 的個人化；在 `rateGame` 開頭加 `if (!currentUser) { 提示登入; return; }`；隨機推薦若選 MBTI/桌遊偏好，未登入時可改為提示或改為純隨機。 |
| **待決** | ~~未登入時「隨機推薦」的個人化選項要隱藏、改為純隨機，還是點下去再提示登入？~~ **決議**：點下去再彈出登入提示。 |

### 2. 我的檔案 (profile.html)

| 項目 | 說明 |
|------|------|
| **現狀** | 無 `localUser` 且無 `googleUser` 時呼叫 `showNotLoggedIn()`，只顯示未登入區塊；有 Google 但 DB 無資料則顯示簡化版檔案。 |
| **B 目標** | 未登入也可**進入頁面**，不擋；內容改為說明「登入後可看個人檔案與偏好」＋登入 CTA，不顯示個人數據。 |
| **寫入點** | 本頁主要為讀取（個人資料、清單、推薦）；無明顯「寫入」按鈕需在 B 時特別擋。 |
| **影響** | **UX**：訪客點「我的檔案」不會被整頁擋，可看到說明與登入引導。**資料**：無影響。**實作**：init 時若未登入則不呼叫載入個人資料的 API，改顯示「登入後可看個人檔案」區塊（可沿用現有 `#not-logged-in` 或新區塊）。 |
| **待決** | 未登入時是否要保留「前往分享喜好」等次要 CTA，或只保留「登入」？ |

### 3. 探索清單 (explore-list.html)

| 項目 | 說明 |
|------|------|
| **現狀** | 依 `googleUser` → 查 DB 取得 `dbUserId`；無則顯示「請先登入，再來管理你的桌遊探索清單」，不顯示主內容。 |
| **釐清** | 此頁是**綁在個人底下**的：沒登入就沒有「自己的」清單，不是「點進別人的」清單。因此 B 的合理做法是：未登入時**只顯示說明＋登入 CTA**，不顯示可編輯的清單 UI（避免誤會）。若未來要支援「看別人的探索清單」（例如 `?user=XXX`），再另案討論。 |
| **B 目標（收斂）** | 未登入可**進入頁面**，僅顯示「登入後可管理你的探索清單」＋登入按鈕；不實作「未登入瀏覽空清單」或「未登入看別人清單」。 |
| **決議** | 暫不實作 B 的變更；維持現狀（未登入即顯示請先登入）。 |

### 5. 遊戲列表 (games-list.html)

| 項目 | 說明 |
|------|------|
| **現狀** | 無 `targetUserId` 時用當前登入者；無 `currentUser` 或載入失敗則 `showNotLoggedIn()`。有 `?user=XXX` 時為**查看他人**清單，需能撈該使用者資料。 |
| **釐清** | 「我的」清單綁在個人底下，沒登入就沒有自己的清單。唯一對訪客有意義的是**點進別人的**：`?user=XXX` 時未登入也可唯讀看該玩家清單。 |
| **B 目標（收斂）** | **無 user 參數**：維持現狀，未登入顯示請先登入。**有 user=XXX**：未登入也可進入並顯示該玩家的遊戲清單（唯讀）。 |
| **決議** | 若實作 B 的此項，僅需：有 `user=XXX` 時不要求 currentUser，照常撈該使用者並顯示；無參數時邏輯不變。 |

| 項目 | 說明 |
|------|------|

## 五、後續可實作項目（依討論結果）

1. **桌遊探索**：未登入時不隱藏整頁，顯示清單與卡片，評分按鈕點擊時再檢查登入並提示。
2. **我的檔案**：未登入時顯示說明＋登入 CTA，不擋進入頁面（可選）。
3. **探索清單／我的清單／遊戲列表**：未登入可進入；建立／編輯／刪除（或看自己的清單）時再檢查登入並提示。
4. 其他頁面依決議補齊「寫入前檢查登入」或「進入即檢查」邏輯。
5. 若有「訪客模式」，可統一在 app 或 router 層處理「允許瀏覽／禁止寫入」規則。
